# Aura Salon CRM/POS AI Audit

Date: 2026-05-17

## Executive Summary

Aura already has a broad AI-shaped product surface, but the current implementation is mostly a deterministic salon intelligence engine. It reads real tenant data from SQLite and produces useful business recommendations, but most "AI" outputs are generated by local rules, keyword checks, fixed templates, and simple scoring formulas.

The only direct model call found is in `server/services/ai-assistant.service.js`: when `AI_PROVIDER=openai` and `OPENAI_API_KEY` are configured, the service calls OpenAI Responses API and attaches `modelText` to the rule-generated response. The main response shape still comes from local business logic. No Anthropic provider, prompt registry, AI cache, cost ledger, budget enforcement, embeddings, model registry, or prompt versioning exists yet.

Recommended next step: implement Phase 1 LLM Foundation before adding more AI UI, so every future AI feature shares provider fallback, caching, cost tracking, prompt versioning, and auditability.

- `GET /api/ai/history`
- `POST /api/ai/:type`

- `appointment-booking`
- `upsell`
- `service-recommendation`
- `chatbot`
- `follow-up`
- `review-reply`
- `marketing-caption`
- `analytics-summary`
- `churn-prediction`

| Endpoint / type | LLM, template, or hybrid | Inputs used | Inputs ignored or weakly used | Output source | Persistence |
|---|---|---|---|---|---|
| `GET /api/ai/history` | Data read only | Query passed to repository list | N/A | Existing `ai_interactions` rows | Reads `ai_interactions` |
| `POST /api/ai/appointment-booking` | Hybrid when OpenAI enabled; otherwise rules | `clientId`, `serviceId`, `staffId`, `branchId`, `startAt`, `source`, `walkIn`, `chair`, `prompt`, `confirmBooking` | Natural-language prompt is only keyword matched; no full NLU slot extraction | Local matching creates draft/appointment; optional OpenAI adds `modelText` only | Creates `ai_interactions`; may create appointment and notification |
| `POST /api/ai/upsell` | Hybrid wrapper, rule-generated core | `clientId`, `serviceId`, `prompt`, `concern` | No co-purchase model; no client similarity model | Service add-ons, top in-stock retail products, membership rule | Creates `ai_interactions` |

| `POST /api/ai/chatbot` | Hybrid wrapper, rule-generated core | Current dashboard/report context | User question/prompt is not meaningfully parsed | Fixed answer from revenue/bookings/pending/low stock | Creates `ai_interactions` |

### `ai_interactions` Population

`ai_interactions` is populated for every successful `POST /api/ai/:type` call through `persistInteraction()`.

Fields saved:
- `tenantId`
- `branchId`
- `clientId`
- `appointmentId`
- `type`
- `prompt`
- `input`
- compact `context`
- `output`
- `actions`
- `model`

- `provider`
- `model_tier`
- `prompt_version`
- `input_tokens`
- `output_tokens`
- `cost_usd`
- `latency_ms`
- `request_id`
- `cached`
- `fallback_reason`
- `error_code`

## WhatsApp AI Readiness

Current WhatsApp automation is useful but not an AI agent yet.

| Capability | Current state | Gap |
|---|---|---|
| Intent detection | Regex/keyword matching in `detectIntent()` | Needs LLM structured JSON intent detection with Hindi/Hinglish |
| Lead scoring | Rule formula in `scoreLead()` | Needs learned conversion propensity later |
| Auto reply | Fixed templates in `sendAutoReply()` | Needs dialog state and same-language response generation |
| Booking tools | None in WhatsApp service | Needs `search_available_slots`, `create_booking`, `reschedule`, `cancel` tool orchestration |
| Conversation state | Thread metadata only | Needs persistent `whatsapp_conversation_state` |
| Human handoff | Exists via `whatsapp_handoffs` | Good base; should be integrated as AI escape path |
| End-to-end booking | Not implemented | AI currently asks staff/customer for details but does not complete booking autonomously |

## Customer 360 AI Readiness

Current Customer 360 reads real client data and computes:
- lifetime value
- average spend
- inactive days
- favorite service
- preferred staff
- outstanding balance
- membership status
- rule-based risk score
- rule-based next-best-action
- timeline events

This is a strong data foundation, but not yet ML/LLM-powered. Churn risk is formula-based, next-best-action uses simple if/else rules, and explanations are fixed strings.

## Tenant And Security Findings

Good foundations:
- Routes are mounted under both `/api` and `/api/v1`.
- `/api/v1` routes are JWT protected.
- Web `/api` routes use request context headers and RBAC.
- Services generally call `tenantService.accessScope()` and assert branch access where applicable.
- AI/marketing outputs persist with tenant scoping.

Risks to address in Phase 1:
- OpenAI prompt currently receives compact context, but no PII redaction option exists.
- API key is not logged in code reviewed, but provider abstraction should enforce key redaction centrally.
- No per-tenant AI rate limit or budget.
- No token/cost ledger for chargeback.
- No prompt/version audit trail.

## Phase 1 Implementation Readiness

Recommended minimal Phase 1 slice:

1. Add `server/config/aiTasks.js`.
2. Add `server/services/ai/llmProvider.js`.
3. Add prompt registry folder `server/services/ai/prompts/`.
4. Add idempotent DB migrations for `ai_response_cache` and `ai_cost_ledger`.
5. Extend `ai_interactions` metadata without breaking existing rows.
6. Convert only one low-risk endpoint first, ideally `review-reply`, to prove provider/cache/cost/fallback.
7. Add tests for fallback, cache hit, cost ledger, timeout, and budget enforcement.
8. Then migrate remaining `/api/ai/:type` workflows.

Do not start RAG, WhatsApp agent, ML models, or AI frontend pages until the shared provider foundation is stable.

## Decision

Phase 0 is complete. The project is ready for Phase 1 LLM Foundation, but implementation should be split into small commits. The current system is valuable as a deterministic fallback and should not be deleted.
